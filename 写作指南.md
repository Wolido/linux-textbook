# 《Linux操作系统与数据库教材》写作指南

本指南面向所有参与本项目写作、审阅和维护的作者。它规定了教材的写作风格、HTML 技术规范、图片标准和审阅流程，目的是让全书保持一致、准确、易读。

---

## 1. 项目定位与目标读者

### 1.1 这本书写给谁看

本教材主要面向**人工智能与材料科学等理工科领域的学习者**，尤其是已经习惯图形界面、但对命令行、服务器和数据库知之甚少的初学者。书中会用材料学、AI 训练、科学计算等场景做例子，帮助读者把 Linux 和数据库与自己的专业工作联系起来。

如果读者来自其他专业，或者只是对 Linux 与数据库感兴趣，书中的基础内容同样适用。

### 1.2 教学目标

- 让读者理解 Linux 操作系统和数据库在 AI 基础设施中的位置。
- 掌握足够的命令行操作、环境配置、软件安装和网络知识，能够独立使用 Linux 服务器。
- 理解数据库的基本概念，能使用 SQL 完成常见的数据查询与管理任务。
- 为后续学习 Docker、云计算、大数据和 AI 训练平台打下基础。

### 1.3 与市面其他教材的区别

- **面向场景，不面向考证**：不是为了应付某个考试，而是为了解决材料、AI、科研中真实会遇到的问题。
- **讲故事，不讲手册**：从历史、设计和工程取舍讲起，而不是罗列命令参数。
- **纯静态 HTML，无构建依赖**：每一页都是可直接打开的 HTML，便于长期维护和离线阅读。
- **持续更新**：章节按写作进度逐步发布，目录中会标注“已更新 / 待更新”。

## 2. 写作风格与语气

### 2.1 像书，不是像文档

读者打开的是“教材”，不是 API 参考手册。因此：

- 要有叙事感，按“为什么出现 → 它是什么 → 怎么用”的顺序展开。
- 可以用历史故事、人物、工程困境引出技术概念。
- 避免“1. 简介、2. 安装、3. 配置、4. 示例”这种文档式八股结构。

**示例**：

> 不好：
> 1. Unix 是什么
> 2. Linux 是什么
> 3. 两者区别
> 4. 本章小结

> 更好：
> “虽然这本书是讲 Linux 的，但故事却要从 Unix 讲起，甚至从 Unix 之前一款已经淹没在历史长河中的操作系统讲起。”（章节开头示例）

### 2.2 口语化、易懂、顺畅

- 使用短句和主动语态。
- 遇到复杂概念时，先用日常比喻解释，再给出准确描述。
- 不要把多个从句套在一起。

### 2.3 比喻与案例

比喻要贴近读者熟悉的事物：

- “内核是硬件与软件之间的唯一翻译官。”
- “包管理器就像操作系统的应用商店。”
- “Debian 系与 Red Hat 系是两个家族，说的是不同的方言。”

案例要具体、可复现，优先使用书中已经涉及的真实发行版、工具和场景。

### 2.4 专业但不冷漠

- 对初学者友好，不炫耀术语。
- 遇到困难或易混淆的地方，主动停下来解释。
- 可以适度使用“你可能会问”“要我说”“千万别把它当成……”等对话式表达。

### 2.5 去 AI 味

避免以下高频 AI 写作痕迹：

- 过度使用“ delve / crucial / pivotal / vibrant / underscore / testament / 归根结底 / 不容忽视 / 具有深远意义”。
- 三段式排比、空洞总结。
- 每一节结尾都硬凑“总之”“综上所述”。
- 滥用 em dash 和长难句。

如果某一节已经说完，不必强行加一段总结。

## 3. 内容组织方式

### 3.1 章节结构

```
h1：章标题
  h2：大节
    h3：小节
      h4：子小节
```

- `h1` 只出现一次，作为章标题。
- `h2` 是章内主要板块。
- `h3`、`h4` 用于细化论述，不要跳级使用。
- 每个 `h2`、`h3`、`h4` 都必须设置 `id`；其中 `h2` 用于侧边栏导航，`h3`、`h4` 主要用于页面内锚点和内部链接。

### 3.2 每章开头与结尾

**开头建议**：

- 用一句话点明本章要解决的问题。
- 可以从一个具体场景、历史事件或读者的疑问切入。
- 不要一上来就定义术语。

**结尾建议**：

- 自然收束，不必机械总结。
- 可以预告下一章，或提出一个让读者思考的问题。
- 页脚“上一章 / 下一章”链接必须正确。

### 3.3 节奏：为什么 → 是什么 → 怎么用

讲解技术概念时，优先遵循以下节奏：

1. **为什么需要它**：解决了什么问题，没有它会怎样。
2. **它是什么**：核心定义、关键组成部分。
3. **怎么用**：命令、操作步骤、实际例子。

例如讲解 `dnf` 时，先讲 rpm 依赖问题为什么会让早期用户头疼，再介绍 yum/dnf 是什么，最后给出具体安装命令。

### 3.4 延伸阅读与案例

- 延伸阅读可以放在相关段落末尾，用 `<a>` 链接到 `case-*.html` 页面。
- 案例页（如 `case-XXXX.html`）用于展开一个独立事件，避免打断主线叙事。

### 3.5 延伸阅读（case 页面）的写作规范

- 延伸阅读页面命名为 `case-XXXX.html`，放在 `cases/` 子目录下。
- 每个 case 页面长度通常控制在 **1000–2000 字**（中文），作为独立小故事。对于综述型、历史型或需要系统梳理某个大主题的延伸阅读，篇幅可以适当突破，但需结构清晰、分段合理，并经过 reviewer 审阅。
- 内容应与正文章节呼应，补充历史背景、真实事件或深度案例，不重复正文已讲内容。
- 使用 `h1` 作为页面标题，`h2` 作为大节，所有 `h2` 必须设置 `id`。
- case 页面的 `h1` 同样建议设置 `id`，与章节页保持一致，便于侧边栏锚点和内部引用。
- 史实、法律事件、年份、人物必须准确；不确定的细节应加限定语或注明来源。
- 叙事型 case 页面以文字为主，不必强制配图；确需配图时遵循第 5 章图片规范。
- 在正文章节中引用 case 页面时，用自然语句引入链接，不要生硬堆砌。

## 4. HTML 技术规范

### 4.1 基本约束

- 本项目是**纯静态 HTML 站点**，不使用构建工具。
- 章节页放在项目根目录，例如 `chapter-02.html`；延伸阅读 case 页统一放在 `cases/` 子目录下，例如 `cases/case-open-source.html`。
- 资源引用使用**相对路径**，例如 `assets/css/style.css`、`assets/images/example.svg`。

### 4.2 新增章节流程

1. 复制 `template.html`，重命名为目标文件名，例如 `chapter-02.html`。
2. 替换页面 `<title>`、`h1` 标题、正文内容和页脚链接。
3. 为所有 `h2`、`h3`、`h4` 设置 `id`。
4. 同步更新以下位置：
   - `index.html` 全书目录；
   - `preface.html` 侧边栏；
   - 已有章节的侧边栏（如 `chapter-01.html`）；
   - 所有已有 `cases/case-*.html` 页面的侧边栏；
   - `template.html` 与 `cases/template-case.html` 侧边栏；
   - 新增页面自身的侧边栏；
   - 相邻页面的页脚“上一章 / 下一章”链接。

   > 注意：章节侧边栏只需同步 `h2` 级标题链接，`h3`/`h4` 不需加入侧边栏（但仍需保留 `id` 供正文内部锚点使用）。case 页侧边栏中的章节链接需使用 `../chapter-XX.html` 等相对路径。

#### 在已有章节中间插入新章

如果在已有章节之间插入新章（例如在 `chapter-02.html` 与 `chapter-03.html` 之间新增一章），除上述流程外，还需按以下顺序处理重新编号问题：

1. 为新章命名并确定序号，将其插入到目标位置。
2. 同步更新 `index.html` 全书目录中的章节序号、标题和链接。
3. 更新所有页面的侧边栏顺序，包括前后章节、`cases/case-*.html`、`preface.html`、`template.html`、`cases/template-case.html` 等，确保导航顺序与 `index.html` 一致。
4. 更新相邻章节的页脚“上一章 / 下一章”链接，保证章节链连续。
5. 检查并更新后续所有章节的编号预期：文件名中的序号、`index.html` 中的状态标签、正文内对“第 X 章”的指称，以及图片路径中涉及的章节编号。

> 建议按“先改目录与文件名，再改侧边栏与页脚，最后核对编号预期”的顺序执行，避免遗漏。

### 4.3 新增延伸阅读流程

1. 复制 `cases/template-case.html`，重命名为 `cases/case-XXXX.html`。
2. 替换页面 `<title>`、`h1` 标题、正文内容。`<title>` 推荐格式为“延伸阅读：XXX - Linux操作系统与数据库教材”，可参考已有 case 页面。
3. 为所有 `h2` 设置 `id`。
4. 设置页脚链接：case 页面不再串联“上一章 / 下一章”，页脚保留“返回上一页”和“返回目录”两个链接。其中“返回上一页”使用 `javascript:history.back()`，调用浏览器历史返回，不固定指向某一页；“返回目录”指向 `../index.html`。
5. 在 `extended-reading.html` 的 TOC（`<ul class="toc">`）中新增对应条目，链接写为 `cases/case-XXXX.html`。
6. 在相关章节正文中自然插入指向新 case 页面的链接，链接写为 `cases/case-XXXX.html`。
7. 用浏览器直接打开新增页面，检查页面显示与导航链接。

### 4.4 标题 id 规范

- `h2` 的 `id` 用于侧边栏锚点，`h3`/`h4` 的 `id` 用于页面内锚点和内部链接；所有 `id` 必须与导航链接中的 hash 一致。
- 推荐使用中文 id，去掉空格和特殊符号。例如：
  - 标题“Unix、GNU 与 Linux” → `id="unix-gnu与linux"`
  - 标题“红帽系的家谱与时间线” → `id="红帽系的家谱与时间线"`
- 英文缩写保留原样，只去掉空格和中文标点。
- 保持同一章节内 id 命名风格一致，不要混用中英文。

### 4.5 页面模板结构

- 章节页参考根目录 `template.html`；
- 延伸阅读 case 页参考 `cases/template-case.html`（已预填 `../` 前缀路径）。

每个页面至少包含：

```html
<!DOCTYPE html>
<html lang="zh-CN">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>第 X 章 标题 - Linux操作系统与数据库教材</title>
  <link rel="stylesheet" href="assets/css/style.css">
</head>
<body>
  <!-- header、sidebar、main、footer、script -->
</body>
</html>
```

### 4.6 链接与导航检查

- 所有内部链接使用相对路径，例如 `chapter-01.html#debian系`。
- 章节页指向 case 页时使用 `cases/case-XXXX.html`；case 页指向根目录页面时使用 `../chapter-XX.html`、`../index.html` 等。
- 新增或调整章节顺序后，必须检查相邻页面的页脚链接。
- 用浏览器打开页面预览时，点击侧边栏每个链接，确认能正确跳转。
- **侧边栏展开规则**：只有当前所在章节在自身页面展开 `h2` 子链接，其他章节在所有页面均保持为折叠的顶层链接。新增或调整章节后，应逐页检查侧边栏是否符合该规则。

## 5. 图片与示意图规范

### 5.1 存放位置

- 图片统一放在 `assets/images/` 目录树下。
- 旧图可保留在 `assets/images/` 根目录，也可按章节迁移到 `assets/images/chapter-XX/` 子目录；建议后续统一按章节存放。
- 引用路径示例：`assets/images/chapter-01/redhat-logo.svg`（已按章节迁移的旧图）、`assets/images/chapter-02/fig-2.5-prompt-mockup.svg`（新图）。

### 5.2 图片标签

所有图片使用 `<figure>` 包裹，并包含 `alt` 和 `figcaption`：

```html
<figure>
  <img src="assets/images/chapter-01/redhat-logo.svg" alt="Red Hat 的徽标">
  <figcaption>图1.5 Red Hat 的徽标</figcaption>
</figure>
```

新图请按 `assets/images/chapter-XX/fig-X.Y-description.svg` 的规范引用，详见 5.6.4。

`alt` 和 `figcaption` 的描述必须与 SVG 实际画出的内容一致，不能超出图中实际存在的元素。例如，图中只展示了三个文件，就不要写成“多个文件和目录”；图中是 CentOS Stream 徽标，就不要写成“CentOS Linux 徽标”。图文不符会误导读者，也会增加返工成本。

### 5.3 图号规则

- 图号按章节连续编号，例如第一章为“图1.1、图1.2……”。
- 图号与 caption 必须一一对应，新增图片后检查后续编号是否连续。
- 如果在章节中间插入新图片，需要将该位置之后的所有图号依次顺延，确保整章编号连续。

### 5.4 SVG 示意图规范

- 优先使用 SVG 绘制示意图、家谱图、时间线和定位图。
- SVG 宽度控制在 **760px–800px** 以内，适配正文 `max-width: 50rem`（约 800px）。
- 图中元素不能重叠，文字不能溢出边界。
- 在不同浏览器中打开检查渲染效果，尤其是中文标签和特殊符号。
- 使用项目配色：
  - 主色：`#0d47a1`
  - 辅助：`#1565c0`
  - 浅蓝：`#42a5f5`
  - 强调：`#ef5350`

#### 5.4.1 元素布局与间距

- 同层级方框、图标或标签之间要留出足够间距，避免彼此贴在一起。例如图 2.18 中三个匹配文件横向排列时，若框与框之间没有间隙，读者会误以为它们是一个整体。
- 相邻层级之间的间距建议 30px–50px，确保箭头、连线和说明文字都能清晰展示，不被挤压。
- 命令框、说明框、图例等辅助元素不要压在主图内容上方，也不要被主图遮挡。放置前检查整张图的视觉层级，必要时扩大画布或调整辅助元素位置。

#### 5.4.2 箭头与连线

- 箭头推荐使用 SVG `<marker>` 配合 `marker-end` 实现，方向清晰且便于统一维护。图 2.18 的通配符匹配示意最初用手绘填充多边形画箭头，指向含糊；改为 `<marker>` 后，三个箭头分别精确指向三个匹配文件。
- 箭头必须明确指向目标元素，多目标场景中每条箭头都要让读者一眼看出“谁指向谁”。如果多个箭头从同一点出发，可用轻微弧线或错位避免重叠。
- 同一图中出现多种线型时，要明确分工：黑色实线表示目录结构或固定层级关系，蓝色虚线表示搜索路径、流程或命中结果。图 2.19 中黑色目录树连线与蓝色虚线箭头最初重叠在 project → 子目录之间，语义混淆；后来把蓝色虚线改为只指向实际命中的 `.log` 文件。
- 当图中同时存在结构线、流程线、命中线等多种语义时，应在角落或底部添加图例，说明每种线型的含义。

#### 5.4.3 多 terminal 布局

- 一个 terminal 窗口内不要左右并排展示多个命令输出。图 2.17 隐藏文件示意最初把两个命令输出塞在一个 terminal 里左右分栏，结果右侧内容溢出；应拆成两个独立 terminal。
- 多个 terminal 并排时，彼此之间要留出间距，建议 ≥30px，避免窗口贴在一起被误认为是同一个 terminal。
- terminal 标题栏高度要适当，建议 ≥36px，左侧红 / 黄 / 绿三个圆点在标题栏内垂直居中，不要贴顶或贴底。

### 5.5 Logo 使用

- 发行版 logo 优先使用官方 SVG 或 PNG。
- `alt` 和 `figcaption` 必须准确描述图片内容。

**示例**：

- 图片是 Red Hat 公司徽标 → 不要写成“RHEL 徽标”。
- 图片是 CentOS Stream 徽标 → 不要写成“CentOS Linux 徽标”。

### 5.6 终端画面的三种呈现方式

教材中经常需要展示命令输入和终端输出。根据场景复杂程度，可选用以下三种方式，优先顺序为：**CSS 样式化代码块 > SVG 终端窗口图 > 真实截图**。

#### 5.6.1 三种方式的选用原则

| 方式 | 适用场景 | 优点 | 缺点 |
|---|---|---|---|
| CSS 样式化代码块 | 普通命令输入与文本输出 | 可搜索、可复制、维护简单 | 无法表现窗口 UI、光标、交互状态 |
| SVG 终端窗口图 | 需要展示提示符拆解、窗口界面、命令高亮、对比效果 | 矢量清晰、风格统一、可控制脱敏 | 制作成本高于代码块 |
| 真实截图 | 必须展示真实系统界面、复杂交互、特定渲染效果 | 最真实 | 需要环境、涉及隐私脱敏、维护困难 |

如果只是让读者跟着敲命令，优先用样式化代码块；如果需要拆解提示符结构或强调窗口元素，再用 SVG 终端窗口图；真实截图只在必要时使用。

#### 5.6.2 SVG 终端窗口图的风格规范

- **外框**：macOS 风格终端窗口：圆角矩形，顶部深灰色标题栏，左侧红 / 黄 / 绿三个小圆点（红 `#ff5f56`、黄 `#ffbd2e`、绿 `#27c93f`），标题栏中间写窗口标题（如 `Terminal — bash`）。标题栏高度建议 ≥36px，三个圆点在标题栏内垂直居中。
- **背景**：深色，推荐 `#1e1e1e` 或 `#2d2d2d`。
- **文字颜色**：
  - 提示符：`#4caf50`（绿色）
  - 命令：`#ffffff`（白色）
  - 输出：`#e0e0e0`（浅灰）
  - 高亮/标注：项目强调色 `#ef5350` 或 `#42a5f5`
- **字体**：系统等宽字体栈，如 `ui-monospace, SFMono-Regular, Menlo, Monaco, Consolas, 'Courier New', monospace`。若系统已安装 JetBrains Mono、Fira Code 等编程字体，可优先使用。
- 标题栏、标注框等非终端文字使用系统无衬线字体栈：`system-ui, -apple-system, BlinkMacSystemFont, 'Segoe UI', sans-serif`。
- **尺寸**：宽度 760–800 px，高度根据内容自适应。
- **内容**：命令、输出、光标、标注等元素应清晰可辨。

已验证样例见 `assets/images/chapter-02/fig-2.5-prompt-mockup.svg`。

#### 5.6.3 制作流程

1. 确定需要呈现的内容：命令、输出、要标注的关键点。
2. 委托 diagrammer 子 agent 按上述风格绘制 SVG。
3. SVG 源文件保存到 `assets/images/chapter-XX/`。
4. HTML 中直接嵌入 SVG，使用 `<img>` 标签引用：
   ```html
   <figure>
     <img src="assets/images/chapter-02/fig-2.5-prompt-mockup.svg" alt="bash 提示符结构示意图">
     <figcaption>图2.5 bash 提示符结构</figcaption>
   </figure>
   ```
5. 本书不再强制要求 PNG fallback。现代浏览器对 SVG 的支持已足够成熟，直接使用 `<img src="...svg">` 即可满足阅读需求。
6. 如确有特殊兼容需求（例如目标阅读器明确不支持 SVG），可另行生成 PNG 作为补充，但不作为默认规范。需要生成 PNG 时，可使用：
   ```bash
   rsvg-convert -w 1560 input.svg -o output.png
   ```
   其中 `-w 1560` 表示输出宽度为 1560 px（约为 800 px 原始宽度的 2× Retina 分辨率），高度自动按比例缩放。

#### 5.6.4 命名与存放

本书图片逐步采用按章分子目录的存放方式。旧图可保留在 `assets/images/` 根目录或迁移到 `assets/images/chapter-XX/`，新图建议按 `assets/images/chapter-XX/` 存放。

- 终端 SVG 存放在 `assets/images/chapter-XX/` 子目录下。
- 文件名格式为 `fig-X.Y-description.svg`（**不包含章节前缀**，因为目录已经是 `chapter-XX`）。
- 例如：`assets/images/chapter-02/fig-2.5-prompt-mockup.svg`。
- PNG（如有）：`fig-X.Y-description.png`，与对应 SVG 同名同目录。

#### 5.6.5 多个 terminal 并排布局

- 一个 terminal 窗口内不要左右并排展示多个命令输出。例如图 2.17 隐藏文件示意最初把两个命令输出塞在一个 terminal 里左右分栏，结果右侧内容溢出；应拆成两个独立 terminal。
- 多个 terminal 并排时，彼此之间要留出间距，建议 ≥30px，避免窗口贴在一起被误认为是同一个 terminal。
- 每个 terminal 的标题栏高度建议 ≥36px，左侧红 / 黄 / 绿三个圆点在标题栏内垂直居中。

### 5.6.6 安全与隐私

- SVG 终端窗口图中应使用虚构用户名、主机名、路径。
- 避免出现真实 IP、密码、密钥、个人身份信息。
- 真实截图必须脱敏后才能使用。

## 6. 史实与准确性

### 6.1 技术史实必须准确

涉及年份、人物、公司、法律事件时，必须做到有据可查：

- 林纳斯·托瓦兹发布 Linux 0.01 的日期：1991 年 9 月 17 日。
- GNU 计划发起时间：1984 年。
- GPL 协议发布时间：1989 年，GPLv2 定稿于 1991 年。
- Red Hat 公司成立：1993 年。

对于延伸阅读中涉及的法律事件、商业决策和争议性历史（例如 Unix 诉讼、开源许可证纠纷、公司并购与项目转向），除了要核对年份和人名，还应尽量标明信息来源或保留可核查的公开资料线索，避免写成无从验证的传闻。

### 6.2 不确定的表述写得更通用

如果某个细节无法确认，不要凭记忆下结论。可以改为：

- “具体是谁提出了这个名称尚无定论。”
- “根据公开资料，这一比例在 55%～60% 之间。”
- “截至 2026 年的数据显示……”

对于快速变化的领域（如人工智能大模型、市场份额、漏洞统计等），建议标注“截至本文写作时”或具体日期，并预留后续复核机制，避免教材内容很快过时。

### 6.3 数据来源

涉及市场份额、用户数量、漏洞统计等具体数据时，尽量注明：

- 统计机构或报告名称。
- 统计时间。
- 数据范围。

例如：

> “根据 2024 年 11 月的报告，Linux 全球服务器操作系统市场份额约为 62.7%。”

### 6.4 避免绝对化

少用“ always / never / 绝对 / 一定”。技术世界充满例外，优先使用“通常”“在大多数情况下”“截至……”等限定语。

## 7. 代码与命令

### 7.1 行内代码

命令、文件名、函数名、路径等使用 `<code>` 包裹：

```html
<p>使用 <code>dnf</code> 安装软件包。</p>
<p>配置文件位于 <code>/etc/hosts</code>。</p>
```

### 7.2 代码块

较长的命令、配置或多行输出使用 `<pre><code>`：

```html
<pre><code>sudo dnf install nginx
sudo systemctl enable nginx
sudo systemctl start nginx</code></pre>
```

注意：代码块中若出现 `<` 或 `>`，必须写成 `&lt;` / `&gt;`，否则在 HTML 中会被解析为标签，导致显示异常。常见场景包括：

- Git 冲突标记，如 `&lt;&lt;&lt;&lt;&lt;&lt;&lt; HEAD`；
- HTTP 响应体中的 `&lt;html&gt;`、`&lt;body&gt;` 等标签，应写为 `&lt;html&gt;`、`&lt;body&gt;`；
- 命令输出中的尖括号或重定向符号，如 `cat &lt; input.txt`。

### 7.3 terminal 代码块的样式规范

本书使用深色背景模拟终端的 `terminal` 代码块展示命令输入与输出。HTML 结构应统一为：

```html
<pre><code class="terminal">
<span class="prompt">$ </span><span class="command">ls -la /var/log</span>
<span class="output">total 128
drwxr-xr-x  1 root root ...</span>
</code></pre>
```

其中：

- `pre` 包裹整个代码块；
- `code` 设置 `class="terminal"`，用于触发终端样式；
- `span class="prompt"` 表示提示符；
- `span class="command"` 表示用户输入的命令；
- `span class="output"` 表示命令输出。

这样结构化的好处是样式可统一调整，也便于后续维护与辅助阅读。

#### 长命令右侧保留呼吸空间

终端代码块背景较深，命令行很长时会触发横向滚动。如果只在 `pre` 上设置右侧 `padding`，滚动到最右端时文字仍会紧贴右边缘——因为 `padding` 不会随内容一起滚动。

正确做法是把右侧内边距放在 `code.terminal` 内部，并让它成为可滚动内容的一部分：

```css
pre:has(code.terminal) {
  padding: 1rem 0 1rem 1rem;
}

pre code.terminal {
  display: inline-block;
  vertical-align: top;
  min-width: 100%;
  padding: 0 1rem 0 0;
  line-height: 1.6;
  white-space: pre;
}
```

要点：

- `pre:has(code.terminal)` 的右侧 `padding` 置为 `0`，避免和内部滚动区域重复。
- `pre code.terminal` 使用 `display: inline-block; vertical-align: top; min-width: 100%`，保证短命令时仍横向铺满容器。
- `code.terminal` 的 `padding-right: 1rem` 随内容一起滚动，长命令滚动到最右端时仍有 ≥1rem 的呼吸空间。

### 7.4 示例要实用、可复现

- 命令尽量给出完整形式，避免让读者猜参数。
- 如果示例需要前置条件（如已安装某工具、已配置 sudo），提前说明。
- 不要写已经废弃或不推荐的命令，除非是为了解释历史演变。

### 7.5 命令输出

如果需要展示命令输出，用注释说明这是输出，而不是让读者误以为要输入这些内容。

## 8. 编辑与审阅流程

### 8.1 一次只写一个主要小节

不要一次性写完一整章再提交。推荐按 `h2` 或 `h3` 单元推进，这样便于审阅、修改和回滚。

### 8.2 本地验证

每完成一个主要小节，直接用浏览器打开对应的 HTML 文件检查。项目为纯静态站点，所有链接使用相对路径，通过 `file://` 协议即可离线浏览，无需启动本地服务。

### 8.3 检查清单

- [ ] HTML 结构完整，无未闭合标签。
- [ ] 所有 `h2`、`h3`、`h4` 都有 `id`（`h3`/`h4` 用于页面内锚点和内部链接）。
- [ ] 侧边栏导航顺序与 `index.html` 一致，且章节页侧边栏只展示到 `h2`。
- [ ] 页脚“上一章 / 下一章”链接正确。
- [ ] 图片路径正确，`alt` 和 `figcaption` 齐全。
- [ ] SVG 无元素重叠、文字溢出。
- [ ] 代码示例可直接复制运行或已注明前置条件。
- [ ] 历史数据、年份、人名已核对。

### 8.4 Reviewer 审阅

- 每个主要修改必须经过至少一位 reviewer 审阅。
- 审阅重点：事实准确性、叙事流畅性、HTML 规范、图片标注、链接正确性。
- 重大问题修改后需重新审阅。

### 8.5 子 agent 执行边界

子 agent（如 coder、writer）用于协助完成代码、文本、HTML/CSS 的修改、语法检查和路径一致性验证。在执行过程中，子 agent **不代替人执行浏览器渲染验证**，不强制在浏览器中打开页面或调用浏览器截图。

- 子 agent 负责：文本、代码、HTML/CSS 的修改与静态检查；路径、链接、标签一致性验证；语法与格式问题的修复。
- 子 agent 不负责：在浏览器中打开页面预览、验证最终视觉渲染效果。
- 本地浏览器预览和渲染效果由作者或审阅人在本地完成，并作为最终验收的一部分。

本规范是对 8.2 “本地验证”的补充：本地验证仍由人完成，子 agent 不必重复执行。

## 9. 红帽系写作中的具体教训

以下以第一章红帽系发行版的写作为例，总结的经验后续章节可引以为戒。

### 9.1 不要写成文档式 1、2、3、4 固定格式

最初如果用“1. 简介、2. RHEL、3. Fedora、4. CentOS……”来罗列，会让读者昏昏欲睡。改成“从一顶红帽子说起”“CentOS：那个免费的午餐变了”这样的小标题后，阅读体验明显提升。

### 9.2 用示意图表达关系比纯文字更有效

Red Hat 系各发行版之间的上下游关系和历史变迁，用文字描述容易绕。改用两张 SVG 图——家谱图和时间线——后，读者一目了然。如果概念之间还有明显的“光谱式”差异（例如稳定 vs 前卫），也可以考虑增加定位图，但要避免信息重复。

### 9.3 发行版家谱、时间线是很好的辅助工具

- **家谱图**：展示 Fedora → RHEL → CentOS / Rocky Linux / AlmaLinux 的衍生关系。
- **时间线**：标注公司成立、版本发布、CentOS 转向等关键事件。

后续章节遇到类似“多个项目并行发展”的主题时，优先画这两类图。如果各项目还有明显的定位差异（例如稳定 vs 前卫），可以再补充定位图，但不要为了画图而画图。

### 9.4 写历史典故时注意事实准确性

Red Hat 名字的来源、CentOS Linux 的结束、Rocky Linux 的命名缘由等，都需要核对多个来源。不能仅凭记忆写“据说”“好像”。

### 9.5 SVG 一定要检查元素重叠和文字溢出

在 800px 宽度下看起来正常的 SVG，缩放到 375px 手机屏幕时可能出现文字换行或元素重叠。必须在不同宽度下检查，必要时为移动端调整布局。

### 9.6 Logo caption 要和实际图片内容一致

第一张红帽 logo 用的是 Red Hat 公司徽标，caption 就不能写成“RHEL 徽标”。CentOS Stream 的 logo 也不能写成“CentOS Linux”。图片与文字必须严格对应，避免误导读者。

## 10. 延伸阅读写作中的具体教训

在逐步扩展延伸阅读体系、新增多个 case 页面时，积累了以下经验，后续在 `cases/` 目录下新增 case 页面时应引以为戒。

### 10.1 不要硬凑图片

历史/法律叙事型 case 页面，文字能把事情讲清楚就不必加图。不要为了“图文并茂”而强行插入与内容关联不大的图片。

### 10.2 史实必须可核查

涉及法律诉讼、公司决策、年份时，不要凭记忆写“据说”“好像”。每个关键事实都应能追溯到公开资料。

### 10.3 控制篇幅

case 页面通常是独立小故事，不要写成完整章节。把最精彩或最关键的部分讲清楚即可，避免面面俱到。但对于综述型、历史型或需要系统梳理某个大主题的延伸阅读，篇幅可以适当突破，前提是结构清晰、分段合理，并经过 reviewer 审阅。

### 10.4 与正文做好衔接

case 页面要在正文章节中被自然引用，形成“主线 + 支线”的阅读体验。不要让它成为孤立的、没人知道存在的页面。

### 10.5 同步范围比章节更大

新增 case 页面时，页脚不再串联“上一章 / 下一章”，保留“返回上一页”和“返回目录”两个链接。其中“返回上一页”使用 `javascript:history.back()`，调用浏览器历史返回，不固定指向某一页；“返回目录”指向 `../index.html`。无需再更新相邻 case 页面之间的互链。

## 11. 常用术语与表述约定

为保证全书术语一致，约定如下：

| 中文表述 | 使用场景 | 说明 |
|---|---|---|
| Linux 发行版 / 发行版 | 通用 | 指以 Linux 内核为核心的完整操作系统发行版本，如 Ubuntu、RHEL、Fedora。 |
| 内核 / Kernel | 通用 | 指 Linux Kernel。首次出现可写“Linux 内核（Kernel）”，后续视上下文选择其一。 |
| 包管理工具 / 包管理器 | 通用 | 两者可互换，但同一章内保持一致。 |
| 命令行 / 终端 / Shell | 通用 | “命令行”强调界面形态；“终端”强调程序窗口；“Shell”强调命令解释器本身。 |
| 开源 / 自由软件 / 开源协议 | 通用 | “自由软件”强调自由使用权；“开源”强调源代码公开；“开源协议”指 GPL、MIT、BSD 等许可证。 |
| GNU/Linux | 必要时 | 强调完整操作系统由 GNU 工具链和 Linux 内核共同构成时使用。 |
| 上游 / 下游 | 发行版关系 | Fedora 是 RHEL 的上游；CentOS Stream、Rocky Linux、AlmaLinux 是 RHEL 的下游。 |
| 长期支持 | 发行版 | 指 LTS（Long Term Support）。首次出现可写“长期支持（LTS）”。 |
| 二进制兼容 | 发行版 | 指软件包无需重新编译即可在不同发行版上运行。 |
| 仓库 / 软件仓库 | 包管理 | 指存储软件包的服务器或镜像。 |
| 桌面环境 | 通用 | 如 GNOME、KDE Plasma、Cinnamon。 |
| 容器 | Docker 章节 | 指 Container，如 Docker 容器。 |
| 数据库 / Database | 通用 | 指 Database。首次出现可写“数据库（Database）”。 |
| SQL / NoSQL | 数据库 | SQL 指结构化查询语言，也泛指关系型数据库；NoSQL 指非关系型数据库。 |

### 11.1 大小写约定

- 专有名词保持官方大小写：`Linux`、`Unix`、`GNU`、`GPL`、`RHEL`、`Fedora`、`Ubuntu`、`Docker`、`SQL`、`Python`。
- 普通术语在句中不大写，例如“发行版”“内核”“命令行”。

### 11.2 中英文混排

- 英文术语与中文之间保留一个空格，例如“使用 dnf 安装软件”。
- 标点使用中文全角标点，但代码中的标点保持原样。

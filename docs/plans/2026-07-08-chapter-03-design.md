## 详细大纲

### 1. 为什么 Linux 要区分用户？

#### 1.1 多用户：一台机器上的多个工作台

- **核心问题**：为什么同一台 Linux 服务器要分给多个人用？
- **内容要点**：
  - Linux 从设计之初就是多用户操作系统，每个用户拥有独立的家目录和配置。
  - 实验室服务器、云主机、HPC 集群通常都是多用户环境。
  - 区分用户能保护个人数据，也能防止一个人的误操作影响整个系统。
- **涉及命令/概念**：用户（user）、家目录、`/home`、`/etc/passwd`（仅概念提及）。
- **示例场景**：
  - 学生 A 和学生 B 同时登录同一台服务器，各自在 `/home/alice` 和 `/home/bob` 下工作，互不影响。
- **配图建议**：图 3.1 SVG 示意图，展示一台服务器上多个用户的家目录相互隔离。

#### 1.2 用户与组：个人身份与团队身份

- **核心问题**：用户和组是什么关系？
- **内容要点**：
  - 每个用户至少属于一个组，通常创建用户时会自动创建一个同名组。
  - 组用于批量管理权限，例如一个课题组共享一个数据目录。
  - 用 `id` 查看当前用户的 UID、GID 和所属组。
- **涉及命令/概念**：`id`、`UID`、`GID`、用户组（group）、主组、附加组。
- **示例场景**：
  ```bash
  id
  # uid=1000(student) gid=1000(student) groups=1000(student),27(sudo)
  ```
- **配图建议**：图 3.2 SVG 示意图，展示“用户属于组，组拥有共享目录”的关系。

#### 1.3 当前用户是谁？`whoami` 与 `who`

- **核心问题**：怎么确认自己现在以什么身份登录？
- **内容要点**：
  - `whoami` 显示当前用户名。
  - `who` 显示当前登录到系统的用户列表。
  - 在多用户服务器上，先确认身份再操作，是避免误动他人数据的好习惯。
- **涉及命令/概念**：`whoami`、`who`、当前会话。
- **示例场景**：
  ```bash
  whoami
  # student
  who
  # student pts/0  2026-07-08 09:00 (10.0.0.5)
  ```

### 2. root 与 sudo

#### 2.1 root：系统的管理员账户

- **核心问题**：root 用户为什么这么特殊？
- **内容要点**：
  - root 是超级用户，UID 为 0，对系统拥有完全控制权。
  - root 可以修改系统文件、安装软件、创建和删除其他用户。
  - 直接用 root 登录风险极高，日常操作应使用普通用户。
  - 第二章提示符中的 `#` 即表示当前为 root。
- **涉及命令/概念**：root、UID 0、超级用户、系统文件。
- **示例场景**：
  ```bash
  root@gpu-lab01:~#
  # # 号提示符表示当前是 root，操作需格外谨慎
  ```
- **配图建议**：图 3.3 SVG 终端窗口图，对比普通用户 `$` 与 root `#` 提示符。

#### 2.2 `sudo`：临时借一把钥匙

- **核心问题**：普通用户怎么执行需要管理员权限的命令？
- **内容要点**：
  - `sudo`（superuser do）让普通用户以 root 身份执行单条命令。
  - 执行时需要输入当前用户密码（首次执行后会有缓存时间）。
  - 只给真正需要权限的命令加 `sudo`，不要滥用。
- **涉及命令/概念**：`sudo`、权限提升、密码缓存。
- **示例场景**：
  ```bash
  sudo apt update
  # 输入当前用户密码后执行系统更新
  ```
- **配图建议**：图 3.4 SVG 示意图，展示普通用户通过 `sudo` 临时获得 root 权限执行命令。

#### 2.3 `sudo` 的安全习惯

- **核心问题**：用 `sudo` 时要注意什么？
- **内容要点**：
  - 不要对所有命令都加 `sudo`，尤其不要 `sudo rm -rf /`。
  - 复制粘贴网上的命令前，先理解它在做什么。
  - 如果系统提示权限不足，先检查是否真的需要管理员权限，而不是直接 `sudo`。
  - 在共享服务器上，滥用 `sudo` 可能影响他人。
- **涉及命令/概念**：`sudo`、权限最小化、安全操作。
- **示例场景**：
  ```bash
  # 不好的习惯：直接用 sudo 执行本不需要的命令
  sudo ls ~/Documents
  # 好习惯：普通用户权限足够时，不要加 sudo
  ls ~/Documents
  ```

### 3. 文件权限与 `chmod`

#### 3.1 读懂权限字符串：`ls -l` 的输出

- **核心问题**：`ls -l` 前面那一长串字母和横线是什么意思？
- **内容要点**：
  - 权限字符串共 10 个字符：第 1 个表示文件类型，后 9 个分三组，分别表示所有者、所属组、其他用户的读（r）、写（w）、执行（x）权限。
  - 用 `ls -l` 查看权限，用 `ls -ld` 查看目录本身权限。
  - 区分文件权限与目录权限：目录的 `x` 表示能否进入。
- **涉及命令/概念**：`ls -l`、`ls -ld`、权限字符串、`rwx`、所有者、所属组、其他用户。
- **示例场景**：
  ```bash
  ls -l report.txt
  # -rw-r--r-- 1 student student 2048 Jul 8 09:00 report.txt
  ```
- **配图建议**：图 3.5 SVG 示意图，拆解权限字符串的每一位含义。

#### 3.2 修改权限：`chmod` 的数字法与符号法

- **核心问题**：怎么让某个文件可执行，或不让别人读取？
- **内容要点**：
  - 符号法：`chmod u+x file` 给所有者加执行权，`chmod go-w file` 去掉组和其他用户的写权限。
  - 数字法：`chmod 755 file` 对应 `rwxr-xr-x`。
  - 初学者先掌握常见组合：可执行脚本 `755`，普通文件 `644`，共享目录 `775`。
  - 修改前先用 `ls -l` 查看原权限，修改后再确认。
- **涉及命令/概念**：`chmod`、符号法、数字法、`u`/`g`/`o`/`a`、`+`/`-`/`=`。
- **示例场景**：
  ```bash
  chmod +x run.sh
  chmod 644 report.txt
  chmod 755 project_dir
  ```
- **配图建议**：图 3.6 SVG 示意图，对比符号法和数字法。

#### 3.3 所有权：`chown` 与共享目录

- **核心问题**：文件归谁所有？怎么把文件交给别人或某个组？
- **内容要点**：
  - `chown` 修改文件所有者，`chgrp` 修改所属组；通常只有 root 或文件所有者能修改。
  - 共享目录场景：课题组数据目录需要组内可读写。
  - 本节不深入 ACL，仅覆盖最常用场景。
- **涉及命令/概念**：`chown`、`chgrp`、所有者、所属组、共享目录。
- **示例场景**：
  ```bash
  sudo chown alice:lab_group dataset.csv
  ls -l dataset.csv
  # -rw-rw-r-- 1 alice lab_group 4096 Jul 8 10:00 dataset.csv
  ```
- **配图建议**：可用 CSS 样式化代码块配合文字说明，无需新增配图。

#### 3.4 权限实战：一个脚本从“打不开”到“能运行”

- **核心问题**：为什么我刚下载的脚本运行不了？
- **内容要点**：
  - 常见原因：文件没有执行权限，或不在 `PATH` 中。
  - 解决流程：用 `ls -l` 检查权限 → `chmod +x` 添加执行权 → 用 `./script.sh` 执行。
  - 强调不要随意给所有文件 `777` 权限，权限要按需分配。
- **涉及命令/概念**：`ls -l`、`chmod +x`、`./script.sh`、执行权限。
- **示例场景**：
  ```bash
  ls -l run_analysis.sh
  # -rw-r--r-- 1 student student 512 Jul 8 11:00 run_analysis.sh
  chmod +x run_analysis.sh
  ./run_analysis.sh
  ```
- **配图建议**：图 3.7 SVG 流程图，展示“检查权限 → 修改权限 → 执行脚本”的完整过程。

### 4. 用包管理器安装软件

#### 4.1 包管理器是什么？

- **核心问题**：为什么 Linux 不直接下载 `.exe` 安装？
- **内容要点**：
  - Linux 软件通常以“包”的形式分发，包管理器负责下载、安装、升级、卸载和解决依赖。
  - 类比：包管理器就像操作系统的应用商店，但更适合服务器环境。
  - 主要家族：Debian/Ubuntu 系用 `apt`，RHEL/Fedora/Rocky/AlmaLinux 系用 `dnf`/`yum`。
- **涉及命令/概念**：包（package）、包管理器、依赖、仓库、发行版家族。
- **示例场景**：
  - 在 Ubuntu 上安装 Python：
    ```bash
    sudo apt update
    sudo apt install python3
    ```
- **配图建议**：图 3.8 SVG 示意图，展示包管理器从仓库拉取软件包并自动解决依赖。

#### 4.2 Debian/Ubuntu 系：`apt` 常用操作

- **核心问题**：在 Ubuntu 上怎么装软件、卸软件、更新系统？
- **内容要点**：
  - `apt update` 刷新软件包列表，`apt upgrade` 升级已安装软件。
  - `apt install 包名` 安装，`apt remove 包名` 卸载，`apt autoremove` 清理不再需要的依赖。
  - 安装前通常需要 `sudo`。
- **涉及命令/概念**：`apt`、`apt update`、`apt upgrade`、`apt install`、`apt remove`、`apt autoremove`。
- **示例场景**：
  ```bash
  sudo apt update
  sudo apt install htop
  sudo apt remove htop
  sudo apt autoremove
  ```
- **配图建议**：可用 CSS 样式化代码块配合文字说明，无需新增配图。

#### 4.3 RHEL/Fedora 系：`dnf` 与 `yum`

- **核心问题**：在 RHEL/Fedora 上又怎么操作？
- **内容要点**：
  - `dnf` 是新一代包管理器，`yum` 是旧版，两者命令结构相似。
  - 常用操作：`dnf install`、`dnf remove`、`dnf update`、`dnf search`。
  - 与 `apt` 的对应关系：概念一致，命令名称不同。
- **涉及命令/概念**：`dnf`、`yum`、`dnf install`、`dnf remove`、`dnf update`、`dnf search`。
- **示例场景**：
  ```bash
  sudo dnf install htop
  sudo dnf remove htop
  sudo dnf update
  ```

#### 4.4 查找与选择软件包

- **核心问题**：怎么知道要装哪个包？
- **内容要点**：
  - 用 `apt search` 或 `dnf search` 搜索包名。
  - 注意区分主包、开发包（通常带 `-dev` 或 `-devel`）、文档包。
  - 推荐优先从官方仓库安装，谨慎使用第三方仓库。
- **涉及命令/概念**：`apt search`、`dnf search`、开发包、文档包、官方仓库。
- **示例场景**：
  ```bash
  apt search python3-numpy
  dnf search numpy
  ```

### 5. 从源码或其他方式安装

#### 5.1 什么时候需要绕过包管理器？

- **核心问题**：包管理器找不到我要的软件怎么办？
- **内容要点**：
  - 包管理器的软件版本可能太旧，或者根本没有收录某些专业软件。
  - 替代方案：源码编译、预编译二进制、Conda、Docker、官方安装脚本。
  - 选择原则：优先用包管理器，其次用发行版官方推荐的方案，最后才考虑手动编译。
- **涉及命令/概念**：源码编译、预编译二进制、Conda、Docker、安装脚本。
- **示例场景**：
  - 实验室需要特定版本的 PyTorch，而系统仓库版本不满足要求。
- **配图建议**：图 3.9 SVG 决策图，展示“先查包管理器 → 再查 Conda/官方二进制 → 最后源码编译”的选择流程。

#### 5.2 源码编译： `./configure`、`make`、`make install`

- **核心问题**：从源码安装软件的标准流程是什么？
- **内容要点**：
  - 典型三步：`./configure` 检查环境并生成 Makefile，`make` 编译，`sudo make install` 安装。
  - 需要安装 `build-essential`（Debian 系）或 `Development Tools`（RHEL 系）等编译工具。
  - 源码安装风险：卸载麻烦、依赖冲突、可能污染系统目录，建议新手在虚拟环境或容器内练习。
- **涉及命令/概念**：`./configure`、`make`、`make install`、编译器、依赖库、Makefile。
- **示例场景**：
  ```bash
  tar -xzf software-1.0.tar.gz
  cd software-1.0
  ./configure --prefix=/opt/software-1.0
  make
  sudo make install
  ```
- **配图建议**：图 3.10 SVG 流程图，展示源码编译的三步流程。

#### 5.3 Conda 与 Python 环境

- **核心问题**：AI 和科研常用的 Python 库怎么装？
- **内容要点**：
  - Conda 是数据科学和 AI 领域常用的包与环境管理工具。
  - 可创建隔离环境，避免不同项目的库版本冲突。
  - 基本操作：`conda create`、`conda activate`、`conda install`。
  - 与系统包管理器的关系：Conda 管理自己的 Python 和库，不轻易替换系统 Python。
- **涉及命令/概念**：Conda、`conda create`、`conda activate`、`conda install`、`environment.yml`。
- **示例场景**：
  ```bash
  conda create -n ai_lab python=3.11
  conda activate ai_lab
  conda install numpy pandas pytorch
  ```
- **配图建议**：图 3.11 SVG 示意图，展示 Conda 环境相互隔离。

#### 5.4 其他补充方式：AppImage、Docker、官方脚本

- **核心问题**：还有哪些安装方式？各适合什么场景？
- **内容要点**：
  - AppImage：单个可执行文件，无需安装，适合桌面应用快速试用。
  - Docker：容器化运行，隔离环境，适合复杂依赖或跨平台复现。
  - 官方安装脚本：一键安装，但要审查脚本内容，避免运行来历不明的代码。
  - 强调：无论哪种方式，都要从可信来源获取。
- **涉及命令/概念**：AppImage、Docker、容器、安装脚本。
- **示例场景**：
  ```bash
  # Docker 示例：运行一个预配置环境
  docker run -it --rm pytorch/pytorch:latest
  ```

### 6. 运行程序与环境变量

#### 6.1 安装好的程序为什么找不到？

- **核心问题**：我明明装了软件，为什么敲命令说找不到？
- **内容要点**：
  - Shell 通过 `PATH` 环境变量中的目录列表来查找可执行程序。
  - 如果程序不在 `PATH` 中，需要用完整路径调用，如 `/opt/bin/myapp`。
  - 用 `which`、`whereis`、`type` 查看命令位置。
- **涉及命令/概念**：`PATH`、环境变量、`which`、`whereis`、`type`、可执行文件搜索顺序。
- **示例场景**：
  ```bash
  which python3
  # /usr/bin/python3
  echo $PATH
  # /usr/local/bin:/usr/bin:/bin
  ```
- **配图建议**：图 3.12 SVG 示意图，展示 Shell 按 `PATH` 中的目录顺序查找命令。

#### 6.2 查看与设置环境变量

- **核心问题**：怎么查看和修改环境变量？
- **内容要点**：
  - `echo $变量名` 查看单个变量，`env` 查看全部环境变量。
  - 临时设置：`export PATH=$PATH:/opt/bin`（只对当前 Shell 有效）。
  - 永久设置：写入 `~/.bashrc` 或 `~/.bash_profile`，然后 `source` 生效。
  - 区分 Shell 变量与环境变量：只有 `export` 后的变量才会被子进程继承。
- **涉及命令/概念**：`export`、`env`、`echo $VAR`、`source`、`.bashrc`、`.bash_profile`。
- **示例场景**：
  ```bash
  export PATH=$PATH:/opt/software-1.0/bin
  echo $PATH
  source ~/.bashrc
  ```
- **配图建议**：图 3.13 SVG 示意图，展示 `.bashrc` 中的 `export` 如何影响新打开的终端。

#### 6.3 实战：把自定义程序加入 `PATH`

- **核心问题**：怎么让自己安装的软件像系统命令一样随处可调用？
- **内容要点**：
  - 推荐做法：在 `~/bin` 或 `~/.local/bin` 下放置个人脚本，并把该目录加入 `PATH`。
  - 对于源码安装到 `/opt` 的软件，把其 `bin` 目录加入 `PATH`。
  - 修改 `~/.bashrc` 后需 `source` 或重新打开终端。
  - 验证：用 `which` 确认命令已被找到。
- **涉及命令/概念**：`PATH`、`~/.bashrc`、`source`、`which`、用户级安装目录。
- **示例场景**：
  ```bash
  mkdir -p ~/bin
  mv my_script.sh ~/bin/my_script
  chmod +x ~/bin/my_script
  echo 'export PATH=$HOME/bin:$PATH' >> ~/.bashrc
  source ~/.bashrc
  which my_script
  my_script
  ```
- **配图建议**：可用 CSS 样式化代码块配合文字说明，无需新增配图。

#### 6.4 环境变量的常见用途

- **核心问题**：除了 `PATH`，还有哪些常用的环境变量？
- **内容要点**：
  - `HOME`：当前用户家目录。
  - `USER`：当前用户名。
  - `LANG` 和 `LC_ALL`：语言与区域设置。
  - `PYTHONPATH`：Python 查找模块的额外路径（仅作概念提及）。
  - CUDA 相关：`CUDA_HOME`、`LD_LIBRARY_PATH`（仅作概念提及，为后续 AI 章节留伏笔）。
- **涉及命令/概念**：`HOME`、`USER`、`LANG`、`LC_ALL`、`PYTHONPATH`、`LD_LIBRARY_PATH`。
- **示例场景**：
  ```bash
  echo $HOME
  echo $USER
  echo $LANG
  ```

## 风险与注意事项

- **权限示例安全**：涉及 `chmod`、`chown`、`sudo` 的示例需强调“只给必要权限”，避免引导学生养成 `chmod 777` 或滥用 `sudo` 的习惯。
- **包管理命令区分**：`apt` 与 `dnf` 命令结构不同但概念相似，需明确告诉学生“先看自己是什么发行版”。
- **源码安装风险**：提醒学生源码安装可能破坏系统包管理器的依赖关系，建议在个人目录、`~/opt`、`/opt` 或容器内练习。
- **环境变量覆盖**：强调修改 `PATH` 时不要把系统目录误删，建议用 `$PATH:/new/path` 追加而非直接覆盖。
- **Docker 与权限**：运行 Docker 通常需要 `sudo` 或属于 `docker` 组，提醒学生不要在生产服务器上随意运行来历不明的容器。
- **与第二章边界清晰**：本章不再重复讲解 `pwd`、`ls`、`cd`、`cp`、`mv`、`rm`、SSH 基础连接等第二章已覆盖内容，只在使用时自然引用。
